🔷 PydanticAI Worker

A typed, tool-calling autonomous worker built on PydanticAI — self-governed, and autonomously detected by the AWCP radar.

framework: PydanticAI model: Ollama (OpenAI-compat) · llama3.1:8b port: 8102 folder: agents/pydanticai_agent

What it is

An autonomous worker built on PydanticAI, which gives the agent strongly-typed, schema-validated tool calls. It talks to a local Ollama model through Ollama's OpenAI-compatible API (no keys). You give it a goal; it queues it, runs the PydanticAI agent to completion (calling tools as needed), and routes its output through its own governed-write path (risk-tiered, with operator approval for high-risk external writes).

Same self-contained, AWCP-decoupled worker engine as the other agents (sends nothing to AWCP; detected by scan) — distinct here in that PydanticAI provides type-safe tool definitions and runs over the OpenAI-compatible Ollama endpoint.

Architecture — how a goal flows

Features

  • Typed tool calls — PydanticAI validates tool arguments against their signatures.
  • Task queue + autonomous worker — goals processed one at a time, multi-step.
  • Self-governed writes — risk-tiered; high-risk external writes pause for operator approval (no control plane required).
  • Risk tiers + approval — save_artifact medium; external_post high (pauses for approval).
  • Deterministic finalize, decoupled (autonomously detected), task-console UI, free/local model.

How to run

  1. Model backend: 
    ollama serve
ollama pull llama3.1:8b      # first time only
  2. (Optional) control plane for live governance: 
    bash telemetry-included/awcp-mcp-temp-main/scripts/run_awcp.sh
  3. Start this worker (self-bootstraps its venv on first run): 
    bash agents/pydanticai_agent/run.sh
  4. Open → http://localhost:8102 and give it a goal.

Or use the control panel: python3 agents/control_panel.py → http://localhost:8099.

Governance & its relationship to AWCP

This agent is decoupled from the AWCP control plane — it sends nothing to AWCP. Governance is split in two:

  • Agent-local (enforced here): risk tiers + an operator approval step that pauses high-risk external_post writes until approved on this agent's own POST /tasks/{id}/approve endpoint.
  • AWCP-side (observe-only): the radar detects this process autonomously by scanning it and lists it (as quarantined, since the agent declares no control hooks) — it can see the agent but does not gate, approve, or degrade it.
Model note. Tool-calling reliability depends on the local model; llama3.1:8b may emit a tool call as text. The deterministic finalize ensures the governed write still occurs. Point PAI_MODEL at a stronger model for cleaner typed tool calls.

Part of the AWCP agent runtimes. The worker engine is shared in agents/pydanticai_agent/awcp_kit.py; this agent supplies the PydanticAI agent and its tools.
