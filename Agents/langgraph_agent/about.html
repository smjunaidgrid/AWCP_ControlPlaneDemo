🧠 LangGraph Worker

A general-purpose, self-governed autonomous task-worker runtime — autonomously detected by the AWCP radar (it sends nothing to AWCP).

framework: LangGraph (ReAct) model: Ollama · llama3.1:8b port: 8100 folder: agents/langgraph_agent

What it is

This is not a chatbot — it is an autonomous worker in the AWCP sense: an "agent on a runtime" that the control plane observes. You give it a goal; it queues the goal, then executes it in multiple steps using its tools (search the web, do arithmetic), and routes its output through its own governed-write path (risk-tiered, with operator approval for high-risk external writes). It is built on LangGraph's prebuilt ReAct agent over a local Ollama model (free, no API keys).

This agent is fully decoupled from AWCP: it makes no calls to the control plane (no registration, no gate, no signals). The Agent Radar registry detects it autonomously by scanning the running process. Its writes are governed by its own local policy — risk tiers + an operator approval step for high-risk external writes.

Architecture — how a goal flows

Features

  • Task queue + background worker — submit goals; the worker processes them one at a time, autonomously.
  • Multi-step execution — the LangGraph ReAct loop calls tools as needed to reach the goal.
  • Self-governed writes — risk-tiered; high-risk external writes pause for operator approval. Enforced by the agent itself — no control plane required.
  • Risk tiers + approval — save_artifact is a medium local write; external_post is a high external write that pauses for operator approval.
  • Deterministic finalize — the runtime always persists its result through the agent's own governed-write path, even if the local model fails to call the write tools itself.
  • Decoupled & autonomously detected — sends nothing to AWCP; the radar discovers it by scanning the process (observe-only).
  • Built-in task-console UI — submit goals, watch governed steps, approve high-risk writes, read results.
  • Free & local — runs on a local Ollama model; no API keys, no accounts.

How to run

  1. Start the model backend (free, local): 
    ollama serve
ollama pull llama3.1:8b      # first time only
  2. (Optional) start the control plane so governance + the registry are live: 
    bash telemetry-included/awcp-mcp-temp-main/scripts/run_awcp.sh
    The worker still runs without it (writes fail-open), but the gate/approval/degradation only have effect when the radar is up.
  3. Start this worker (self-bootstraps its venv on first run): 
    bash agents/langgraph_agent/run.sh
# to demo the approval flow on every task:
AGENT_FINALIZE_EXTERNAL=true bash agents/langgraph_agent/run.sh
  4. Open the console → http://localhost:8100, type a goal, watch it run.

Or manage all agents from one place: python3 agents/control_panel.py → http://localhost:8099 (start/stop each agent).

Governance & its relationship to AWCP

This agent is decoupled from the AWCP control plane — it sends nothing to AWCP. Governance is split in two:

  • Agent-local (enforced here): risk tiers + an operator approval step that pauses high-risk external_post writes in awaiting_approval until approved on this agent's own POST /tasks/{id}/approve endpoint.
  • AWCP-side (observe-only): the radar detects this process autonomously by scanning it and lists it (as quarantined, since the agent declares no control hooks). AWCP can see the agent but does not gate, approve, or degrade it.
Model note. Tool-calling quality depends on the local model. llama3.1:8b sometimes prints a tool call as text instead of executing it; the deterministic finalize ensures the governed write still happens. A stronger model (e.g. qwen2.5:7b via LG_MODEL) makes the in-task tool calls cleaner.

Part of the AWCP agent runtimes. The queue, worker, governance, approval flow and UI are shared in agents/langgraph_agent/awcp_kit.py; this agent supplies the LangGraph framework agent and its tools.
