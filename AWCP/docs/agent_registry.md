# Agent Workforce Control Plane (AWCP) Registry
## Architectural Documentation & Evolution Roadmap

---

## Executive Summary
The AWCP Registry is the central nervous system of our distributed AI architecture. Its primary job is to catalog available agents and tell external orchestrators (like Temporal) where to route tasks.

The registry has been upgraded from a **Passive Phonebook** to an **Active Admission Controller**. It now uses environment variable introspection and file-level hashing to dynamically assign governance metadata at boot time — with zero hardcoding in individual agent files. Agents requesting write access without telemetry enabled are automatically placed in Quarantine and blocked at the FastAPI layer.

---

## Part 1: What is Created So Far (Phase 1 MVP)

### Registry MVP

- [x] **Stable Identity (Route Hashing)** — Agent IDs are now generated by hashing the agent's route (`agt_{md5(route)[:8]}`). IDs are permanent and do not shift when agents are added or removed.
- [x] **Dynamic Versioning (File Content Hashing)** — The registry reads the physical Python file of each agent's handler using `inspect.getfile()`, hashes its bytes, and assigns a version of the form `1.0.0-auto+{hash[:7]}`. Version changes automatically whenever the agent's source code changes.
- [x] **Zero-Hardcode Ownership tracking** — Owner is resolved from the `AWCP_DEFAULT_OWNER` environment variable, falling back to the machine's OS username via `getpass.getuser()`. No agent file declares an owner.
- [x] **Write Scopes & Feature Flags data contract** — `AgentSpec` exposes `write_scopes: list[str]` and `feature_flags: dict[str, bool]` as optional fields with empty defaults. Individual agent files do not need to set them. The registry merges agent-declared flags with the global `telemetry_enabled` flag resolved from `AWCP_TELEMETRY_ENABLED`.
- [x] **Quarantine Gate (Admission Control based on Telemetry capability)** — At boot, the registry evaluates each agent. If `write_scopes` is non-empty and `AWCP_TELEMETRY_ENABLED=false`, `status` is set to `"quarantined"`. The FastAPI router checks this status on every request and returns `403 Forbidden` before any LLM execution occurs.

### Previously Completed Capabilities
- [x] **Zero-Config Discovery** — `discovery.py` scans the local `agents/` folder and automatically imports agents using Python's `pkgutil`.
- [x] **Immutable Store** — `store.py` keeps an in-memory dictionary of agents populated once at boot.
- [x] **REST API** — `routes.py` exposes `GET /agents` and `GET /agents/{agent_id}` for external orchestrators.
- [x] **Network Visibility** — Exposes fully-qualified public endpoints (e.g., ngrok URLs) via `AWCP_TUNNEL_BASE_URL`.
- [x] **Runtime self-declaration** — Each agent declares its own `runtime` in its `AgentSpec`. No inference logic in the registry.

### Previous Limitations Now Resolved
- ~~Positional IDs shift when agents are added or removed~~ → Stable route-hashed IDs
- ~~Every agent hardcoded to `"1.0"`~~ → Dynamic file-content versioning
- ~~No owner, feature flags, or write scopes~~ → Zero-hardcode governance fields
- ~~No quarantine~~ → Active admission control with 403 enforcement

## Part 2: Remaining Work

### Registry Governance (Not Yet Built)
- [ ] **Per-agent owner override** — Allow `AWCP_OWNER_{AGENT_NAME}` env var to override the default owner per agent.
- [ ] **Observability status tracking** — Track whether an agent has emitted OTel spans, and reflect that in the registry entry.
- [ ] **Policy callbacks** — Hooks the control plane can call to enforce decisions directly on the agent at runtime.
- [ ] **Manual quarantine API** — `PATCH /agents/{agent_id}/status` for operators to quarantine/release agents without restarting.
- [ ] **Persistence** — Registry backed by durable storage rather than in-memory dict.

### Control Plane Primitives (Not Yet Built)
- [ ] **Policy Engine** — Evaluate tool calls against declared write scopes before execution.
- [ ] **Approval tokens** — Expiring, branch-scoped, single-use tokens for high-risk write actions.
- [ ] **Write gating** — Every state-changing tool call checked against policy before it runs.
- [ ] **Degradation ladder** — Progressive autonomy reduction: trace sampling → tighter limits → safer profile → recommendation-only → hard stop.
- [ ] **Replayable Evidence Ledger** — Step history, context hashes, degradation state, and resume pointers.
- [ ] **External agent registration** — Inbound `POST /agents/register` for agents running on remote runtimes.

## Part 3: How to Scale it Up (The Enterprise AWCP Version)

The "Zero-Hardcode" setup follows the **12-Factor App methodology**. By relying on environment variables rather than hardcoded Python strings, the current codebase will seamlessly transition to a mature Cloud/Kubernetes environment with zero rewrites.

### 1. Evolving Ownership via CI/CD
- **Local:** Reads the OS username via `getpass.getuser()`.
- **Enterprise:** The pipeline injects `AWCP_DEFAULT_OWNER="platform-team"`. Python code stays the same.

### 2. Evolving Versioning via GitOps
- **Local:** Hashes the physical file contents.
- **Enterprise:** CI/CD injects the actual Git SHA (`AWCP_GIT_SHA=f8e9a1b`). The registry checks this variable first before falling back to file hashing.

### 3. Evolving Telemetry & Feature Flags
- **Local:** A simple boolean `AWCP_TELEMETRY_ENABLED` env var drives the quarantine gate.
- **Enterprise:** That same boolean acts as the master switch to boot up Datadog, Prometheus, or OpenTelemetry connections. No registry logic changes required.

### 4. Evolving the Registry Store (Persistence)
- **Local:** `store.py` is an in-memory dictionary.
- **Enterprise:** Backed by PostgreSQL/Redis or Kubernetes CRDs. Enables `PATCH /agents/{id}/status` across a global network.

### 5. Evolving Discovery (External Adapters)
- **Local:** `pkgutil` scans the local hard drive for Python files.
- **Enterprise:** An inbound `POST /agents/register` webhook allows agents in Node.js, Go, or Java on remote servers to self-register. The same quarantine logic applies.
