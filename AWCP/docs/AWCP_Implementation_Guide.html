1. What this project is

A small system that runs AI agents and puts a governance layer on top of them.

You type a question. An agent decides whether it needs to look something up on the web. If it does, it runs a web search tool, reads the results, and writes a clean answer. The governance layer sits above all of this and controls what the agent is allowed to do — for example, it can take away the agent's permission to use tools and make it answer from memory only.

There are three agents available: ollama (plain chat), ollama-search (chat + web search), and deepseek / llama-vision (cloud models). The ollama-search agent is the interesting one because it actually uses a tool.

2. The big idea (in one minute)

The system uses Temporal — a workflow engine that runs steps reliably and records every step — as the boss that orchestrates the agent. And it uses an MCP server (Model Context Protocol) as the worker that actually does the small jobs: think, search, write.

You (browser)→ Control API→ Temporal (the boss)→ MCP server (the worker)→ Agent + Tools

The key point: Temporal calls each small job separately, so every step is visible and controllable. This replaced an older design where the agent did everything secretly inside one black box and Temporal only saw the final answer.

3. Change 1 — Temporal drives the tools New work

Before, the whole agent ran inside one web request. Now the steps are split apart so Temporal runs them one by one.

Before vs. after

BeforeAfter
Temporal called one web endpoint. The agent searched and answered secretly inside that call. Temporal just displayed the final result. Temporal calls four small steps itself. It decides when the tool runs. Every step shows up separately and can be governed.

The four steps Temporal now runs

1. get_agent_info→ 2. agent_route→ 3. execute_tool→ 4. agent_generate
  • get_agent_info — Admission check. Is this agent allowed to run, or is it quarantined?
  • agent_route — Thinking step. Does this question need a web search, or can the agent just answer?
  • execute_tool — The tool call. Temporal itself runs the web_search tool. This is the step Temporal now controls.
  • agent_generate — Writing step. Turn the search results (or its own knowledge) into a clean answer.
The governance gate. Between thinking and the tool call, Temporal checks the autonomy profile:
  • active → the tool is allowed → Temporal runs web_search, then writes a grounded answer.
  • recommendation_only → the tool is blocked → Temporal skips the search and the agent answers from memory.
If a run fails in active mode, Temporal automatically retries once in recommendation_only mode. That's the graceful degradation ladder.

How they talk to each other

Each Temporal step opens the MCP server as a small background program (stdio connection), asks it to do one job, and reads the answer back. The MCP server reuses the existing agent code — no agent logic was rewritten, it was just exposed as separate callable tools.

Importantly, the server does not hardcode anything about specific agents. Each agent declares its own behaviour (which model it uses, whether it can use a tool, and which tool). The thinking step and the tool step read those declarations — so brand-new agents work with no server changes. See section 11.

Built on FastMCP. The MCP server uses the FastMCP framework: each tool is a normal Python function marked with @mcp.tool(), and its input schema is generated automatically from the function's type hints — no hand-written schemas. The same file serves both transports: stdio (what Temporal spawns locally) and SSE over HTTP on :8002 (for network sharing). Switching to FastMCP changed none of the tools or behaviour — it just removed boilerplate. One side effect: required arguments are now validated by the framework before a tool runs, instead of being checked by hand inside it.

4. Change 2 — Control surface & cleanup New work

You asked for a place to type inputs and watch the work happen — not the command line — plus a tidy, professional project layout.

What was added

A web control page new

A simple page where you pick an agent, type a prompt, hit Run, and watch the four steps light up live. Shows the answer and badges for status, autonomy, and whether a tool was used.

A control API new

A small bridge. The browser can't talk to Temporal directly, so this service takes your prompt, starts the workflow, and reports live progress back to the page.

One clean virtual environment changed

There used to be two Python environments — one of them empty and unused. Now there is a single .venv with everything in it.

A professional src/ layout changed

All the code moved under one tidy package, src/awcp/. Junk files (empty logs, stray installer, mockup clutter) were cleared out.

Why two Python environments existed (your question): no good reason. One (.venv) was empty and abandoned; the other (mcp_venv) quietly held everything despite its name. They are now merged into a single .venv.

What did not change unchanged

The actual behaviour of the agents, the web-search tool, the registry, and the governance workflow logic is the same as before. The reorg only moved files and updated import paths. Nothing about how an agent thinks or answers was altered.

7. How to run it

You need three things running at once, each in its own terminal. All commands run from the project root.

Prerequisites: Ollama running locally (for the AI models), the Temporal CLI installed, and the project's .venv already built. To rebuild the environment from scratch: 
python3 -m venv .venv
./.venv/bin/pip install -r requirements.txt

Step 1 — Start Temporal (the boss + its dashboard)

temporal server start-dev

This starts the workflow engine on localhost:7233 and its web dashboard on localhost:8233.

Step 2 — Start the worker (does the governed runs)

./scripts/run_worker.sh

This connects to Temporal and waits for jobs. It launches the MCP worker for each step.

Step 3 — Start the control surface (your input page)

./scripts/run_control.sh

Now open http://localhost:8003 in your browser.

Optional extras (not needed for the UI flow):
  • ./scripts/start_mcp.sh — the MCP server's own dashboard + SSE endpoint on :8002.
  • ./scripts/start_server.sh — the older direct REST agent API on :8001.

8. How to use the UI

  1. Open http://localhost:8003.
  2. Pick an agent (use ollama-search to see a tool call happen).
  3. Type a prompt — e.g. "What is the current price of silver per ounce?"
  4. Click Run governed workflow.
  5. Watch the four steps go from pending → scheduled → running → completed.
  6. Read the badges: the final status (SUCCESS / DEGRADED / BLOCKED), the autonomy mode, and whether a tool was used.
  7. Read the answer at the bottom, and click "View full event history in Temporal" to see every detail in the Temporal dashboard.
What "good" looks like: for a price question on ollama-search, all four steps turn green, the status is SUCCESS, tool used is true, and the answer is based on live search results.

9. Troubleshooting

SymptomLikely cause & fix
Run shows FATAL_FAILURE and a "mcp_venv/bin/python not found" type errorAn old worker from a previous session is still running and grabbing jobs. Stop all workers and start one fresh:
pkill -f run_worker then ./scripts/run_worker.sh
UI never leaves RUNNINGThe worker isn't running, or Temporal isn't up. Check Steps 1 and 2.
Answer is empty or oddOllama may not be running, or the web search was rate-limited. Make sure Ollama is up and try again.
"Address already in use" on :7233Temporal is already running — that's fine, just reuse it.
"Agent is in QUARANTINE" / BLOCKEDExpected behaviour: the governance layer refused an under-instrumented agent. Not a bug.

11. Adding a new agent New work

Agents are detected automatically, and each one declares its own behaviour — so you can add a brand-new agent, even a tool-using one, without touching the MCP server or the Temporal workflow.

How detection works

On startup, the registry scans the agents folder and registers any file that defines an AGENT. Nothing is hardcoded — drop a file in, and it appears (in the registry, the MCP server, and the UI's agent dropdown). This is startup-time detection, so a new or removed agent shows up after a restart.

Restart to pick up changes. After adding or removing an agent file, restart the worker (./scripts/run_worker.sh) and the control surface (./scripts/run_control.sh). The MCP worker re-discovers agents every time it runs.

What each agent declares

An agent's AGENT = AgentSpec(...) can include these fields. The control plane reads them to drive the run — no per-agent code lives in the server.

FieldWhat it meansIf left out
name / routeThe agent's id and its REST path.Required.
handlerThe function that runs the agent for the direct REST path.Required.
modelThe model used to write answers.Falls back to a default model.
routerA small function route(prompt) that returns {"action": "SEARCH" or "ANSWER", "search_query": ...} — i.e. does this need a tool?No tool. The agent always answers directly.
toolWhich tool to run when the router says SEARCH (e.g. web_search).Defaults to web_search if a SEARCH is requested.

The rule of thumb

  • Answer-only agent (like ollama): declare a model, no router. Temporal will always route it to ANSWER and skip the tool step.
  • Tool-using agent (like ollama-search): declare a model, a router, and a tool. Temporal will run the tool itself when the router asks for it, then write a grounded answer.
That's the whole point of this change. Previously the server had a special line that only knew about ollama-search. Now the behaviour travels with the agent, so the platform grows by adding files — not by editing the engine.

Minimal example (sketch)

# src/awcp/agents/my_agent.py
def route(prompt):
    # decide if a tool is needed
    return {"action": "SEARCH", "search_query": prompt}

AGENT = AgentSpec(
    name="my-agent",
    route="/chat/my-agent",
    request_model=PromptRequest,
    handler=run,
    runtime="ollama",
    model="llama3.1:8b",   # used to write the answer
    router=route,            # omit this for an answer-only agent
    tool="web_search",     # the tool Temporal will run
)

Restart the worker + control surface, and my-agent shows up in the UI and runs end-to-end — tool calls and all — with no changes to the MCP server or the workflow.
